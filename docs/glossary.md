# Glossary

Domain-specific terms and definitions used throughout the Data Manager project. This glossary supports requirements documentation, design documents, and code comments.

---

## Authentication & Identity

**Authentication (Auth)**
The process of verifying a user's identity through credentials (password, ORCID, personal access token, etc.). See requirements: AUTH-R-01, AUTH-R-02, AUTH-R-03, AUTH-R-04.

**Authorization (Authz)**
The process of determining what authenticated users are allowed to do (access control). See requirements: USER-R-01, USER-R-02.

**Multi-Factor Authentication (MFA)**
A security mechanism requiring multiple verification methods (e.g., password + TOTP code). Currently supports TOTP (Time-based One-Time Password). See requirement: AUTH-R-02.

**TOTP (Time-based One-Time Password)**
RFC 6238 standard for time-synchronized, single-use codes generated by authenticator apps (Google Authenticator, Authy, Microsoft Authenticator). See requirement: AUTH-R-02, AUTH-NFR-05.

**Personal Access Token (PAT)**
An encrypted, user-generated token for programmatic API access without exposing passwords. Valid for 30 days (default, configurable). See requirement: AUTH-R-04.

**ORCID (Open Researcher and Contributor ID)**
Persistent, unique identifier for researchers enabling institutional integration and researcher identity linking. Used for OAuth2/OIDC authentication. See requirement: AUTH-R-03.

**Session**
Authenticated user state maintained across requests via HTTP-only secure cookies. Subject to timeout and invalidation on password reset. See requirements: AUTH-NFR-03, AUTH-NFR-01.

## Project & Data Management

**Project**
Top-level organizational unit containing experiments, samples, measurements, and raw data files. Governed by role-based and ACL-based access control. See requirement: PROJECT-R-01.

**Experiment**
A structured investigation within a project, including experimental design, variables, factors, and sample assignments. See requirement: PROJECT-R-02.

**Sample**
Individual specimen or material being analyzed, linked to experiments and metadata (e.g., organism, tissue, treatment). Can be bulk-imported via XLSX/TSV. See requirement: PROJECT-R-03.

**Measurement**
Analytical result or observation (e.g., NGS sequencing run, proteomics LC-MS/MS analysis). Includes instrument-specific metadata and links to samples. See requirement: PROJECT-R-04.

**Measurement Type**
Category of analytical measurement (NGS, Proteomics, etc.). Each type has its own metadata schema (platform, read count, library type, etc.).

**Batch**
Group of samples or measurements processed together, enabling bulk operations (status update, metadata assignment).

**Raw Data**
Unprocessed output files from analytical instruments (FASTQ, BAM, mzML, etc.). Tracked and made available for download. See requirement: DATA-R-01 (if defined).

## Data Quality & Validation

**Data Validation**
Process of checking uploaded files and user input against schema (required columns, data types) and business rules (cross-references, duplicates). See requirement: QUALITY-R-01.

**Schema Validation**
Verification that uploaded files contain required columns and proper data types. Part of the validation framework.

**Cross-reference Validation**
Ensures referenced entities exist (e.g., sample belongs to project, experiment exists before assigning samples).

**Controlled Vocabulary**
Predefined list of allowed values for a field (e.g., measurement type, organism, tissue type). Values sourced from external terminologies (TIB Terminology Service, OpenBIS).

## FAIR Data & Export

**FAIR Principles**
Framework for research data stewardship: **Findable** (indexed, discoverable), **Accessible** (retrievable by authorized users), **Interoperable** (machine-readable, standard formats), **Reusable** (documented, clear usage rights). See requirement: FAIR-R-01, FAIR-R-02.

**RO-Crate (Research Object Crate)**
Standard package format for research data with structured metadata (ro-crate-metadata.json), files, and provenance information. Conforms to RO-Crate 1.1 specification. See requirements: FAIR-R-01, FAIR-NFR-01.

**JSON-LD (JSON for Linking Data)**
Serialization format for RDF graphs, used in RO-Crate metadata for linked data and semantic interoperability.

**Provenance**
Record of data origin, creation, modifications, and contributors. Included in RO-Crate exports for audit and reproducibility.

**Metadata**
Descriptive information about data (who created it, when, how, what instruments were used, quality metrics, etc.).

## User & Access Management

**Role-Based Access Control (RBAC)**
Access control model where permissions are granted based on user roles (Administrator, Project Manager, Researcher, Viewer). See requirement: USER-R-01.

**Access Control List (ACL)**
Fine-grained permission model enabling project managers to grant granular permissions (View, Edit, Administer) on individual projects without creating new platform roles. See requirement: USER-R-02.

**Permission**
Grant of authority to perform a specific action (create, read, update, delete) on a resource. Assigned via role or ACL.

**Principal**
Entity (user, team, service account) that can be granted permissions. See Spring Security ACL documentation.

---

## Notifications & Communication

**Email Notification**
Asynchronous message sent to user email address for account events (registration confirmation, password reset, project invitation) or critical alerts. See requirement: COMM-R-01.

**In-Application Notification**
Synchronous or quasi-synchronous feedback in the UI (toasts, banners) for operation status, warnings, and errors. See requirement: COMM-R-02.

**Audit Log**
Record of security-relevant events (login attempts, MFA setup, password reset, token usage) with timestamp, IP address, outcome, and context. Retained for 90+ days. See requirement: AUTH-NFR-02.

## Technical & Architecture

**Bounded Context**
Domain-Driven Design concept for a distinct subdomain with its own domain model, language, and organizational boundaries. Data Manager has three main contexts: `identity`, `project-management`, `finances`. See AGENTS.md.

**Domain Event**
Significant occurrence in a bounded context (e.g., UserRegistered, ExperimentCreated) that other contexts may subscribe to. Dispatched in-process via `DomainEventDispatcher` or across contexts via ActiveMQ Artemis.

**Integration Event**
Cross-context event published to Artemis JMS pub-sub for asynchronous communication between bounded contexts.

**OpenBIS**
Enterprise sample and data repository integration. Provides sample management, vocabulary services, and measurement data synchronization. Configured via external service client in `project-management-infrastructure`.

**TIB Terminology Service**
External ontology lookup service for standard terminology (organisms, tissues, assay types, etc.). Supports FAIR data requirements.

**Spring Security ACL**
Framework for Access Control List management in Spring applications. Used for project-level permission granularity. Requires `acl_class`, `acl_sid`, `acl_object_identity`, `acl_entry` database tables.

**Vault (PKCS12 Keystore)**
Encrypted storage for sensitive configuration data (encryption keys for PAT, password reset tokens, TOTP secrets). Protected by a high-entropy password. See AGENTS.md.

**JobRunr**
Background job scheduler and executor. Provides dashboard on port 8000 for job monitoring. Used for scheduled synchronization and export tasks.

**ShedLock**
Distributed lock mechanism preventing duplicate background job execution in multi-instance deployments.

**Vaadin**
Server-side Java UI framework (Flow). Views are `@Route` annotated classes. Theme: `datamanager`. See AGENTS.md.

## Data Governance & Compliance

**GDPR (General Data Protection Regulation)**
EU regulation for data protection and privacy. Data Manager implements GDPR requirements for encryption (Art. 32), audit logging, account recovery, and consent management.

**Research Data Governance**
Institutional policies and best practices for managing research data (retention, access, sharing, preservation).

**Data Use Agreement (DUA)**
Legal contract specifying restrictions and permitted uses of shared data. May include embargoes, publication requirements, institutional affiliations.

**Embargo**
Temporary restriction on data access or publication, after which access becomes available automatically.

**Dataset**
Cohesive collection of data (samples + measurements + metadata) typically exported or published as a unit, often as an RO-Crate.

## External Services & Standards

**OAuth2 / OIDC (OpenID Connect)**
Industry standard for federated authentication and identity management. Data Manager uses ORCID as an OAuth2 provider. See requirement: AUTH-R-03.

**ROR API (Research Organization Registry)**
External API for organization metadata lookup, used for institutional affiliation identification. See AGENTS.md.

**CycloneDX**
Standard format for Software Bill of Materials (SBOM). Used in `sigcheck` Maven profile to generate dependency manifests.

**sigstore/cosign**
OIDC-based artifact signing tool ensuring supply chain security. Signs JAR artifacts; verification prevents tampering. See AGENTS.md.

## Testing & Quality Assurance

**Unit Test**
Fast, isolated test of a single class or method without external dependencies. In Data Manager: Spock `*Spec.groovy` files.

**Integration Test**
Test that verifies multiple components working together, often requiring database or Spring context. In Data Manager: `*IT.java` or `*IT.groovy` files, run under `it` Maven profile.

**Spock Framework**
Groovy-based testing framework for readable, BDD-style test specifications. Preferred for new tests in Data Manager.

**Acceptance Test (from Acceptance Criteria)**
Verification that a Story's acceptance criteria are satisfied. Often manual, but may be automated via Gherkin steps or Spock specs.
